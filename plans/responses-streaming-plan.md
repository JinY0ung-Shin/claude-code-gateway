## Overview

**Goal**: `/v1/responses` 엔드포인트에 OpenAI 공식 Responses API 스펙과 호환되는 SSE 스트리밍을 구현한다.

## Requirements Summary

### Problem Statement
현재 `/v1/responses`는 `stream=true` 요청 시 동작하나, SSE 이벤트 포맷이 OpenAI 공식 스펙과 다수 불일치하여 OpenAI SDK 클라이언트에서 정상 파싱되지 않는다.

### Current Draft Issues (9개)
1. `data: {"event": ...}` 포맷 사용 → 정규 SSE `event: <type>\ndata: <json>\n\n` 필요
2. `response.content_part.delta` 이벤트명 → `response.output_text.delta`
3. `response.done` → `response.completed`
4. `response.in_progress` 이벤트 누락
5. `sequence_number` 필드 누락 (모든 이벤트에 필수)
6. `item_id`, `output_index`, `content_index` 필드 누락
7. `output_item` 키 → `item` 키
8. `content_part` 키 → `part` 키
9. `data: [DONE]` 사용 → 불필요 (response.completed로 종료)

### Scope
**In scope (MVP)**:
- 텍스트 응답 스트리밍 (8개 이벤트)
- 정규 SSE wire format 준수
- `sequence_number` 기반 이벤트 정렬
- `previous_response_id` 체이닝 지원
- 에러 시 `response.failed` 이벤트
- 스트리밍/비스트리밍 최종 결과 일관성

**Out of scope (후속)**:
- Tool call 스트리밍 이벤트 (`response.function_call_arguments.delta` 등)
- Reasoning 이벤트 (`response.reasoning_summary_text.delta` 등)
- Audio/Image 스트리밍 이벤트
- `response.output_text.annotation.added`

### Success Criteria
- [ ] OpenAI Python SDK `client.responses.create(stream=True)` 로 정상 파싱
- [ ] 8개 MVP 이벤트가 올바른 순서로 전송
- [ ] 모든 이벤트에 `sequence_number` 포함
- [ ] 스트리밍 최종 텍스트 == 비스트리밍 응답 텍스트 (parity)
- [ ] `previous_response_id` 체이닝이 스트리밍에서도 동작
- [ ] 에러/중단 시 `response.completed` 미전송, `response.failed` 전송

### Constraints
- 기존 비스트리밍 `/v1/responses` 동작 변경 없음
- 기존 `/v1/chat/completions` 스트리밍 영향 없음
- Claude SDK chunk 추출 로직(`extract_stream_event_delta`) 재사용

### Assumptions
- Claude SDK의 `TOKEN_STREAMING` 이벤트가 텍스트 델타 제공
- OpenAI SDK 클라이언트가 `event:` + `data:` SSE 포맷만 수용

## Architecture

### Approach
기존 chat 스트리밍과 Responses 스트리밍을 **완전 분리된 이벤트 레이어**로 구현. Claude SDK chunk 파싱은 공유하되, SSE envelope 생성은 별도 함수/모델 사용.

```
Claude SDK chunks
    |
    v
extract_stream_event_delta() [공유 - 기존 그대로]
    |
    v
+-- /v1/chat/completions --> make_sse() + stream_chunks()     [기존]
|
+-- /v1/responses        --> make_response_sse() + stream_response_chunks() [수정]
```

### Key Components
- **`ResponseStreamEvent` models** (`response_models.py`): 각 이벤트 타입별 데이터 구조
- **`make_response_sse()`** (`streaming_utils.py`): 정규 SSE 포맷 빌더 (event: + data:)
- **`stream_response_chunks()`** (`streaming_utils.py`): Responses 전용 스트리밍 오케스트레이터
- **`_run_stream()`** (`main.py`): 엔드포인트 스트리밍 제너레이터

### Data Flow
1. 클라이언트 → `POST /v1/responses` with `stream=true`
2. ID 생성 (`resp_*`, `msg_*`) 및 사전 등록
3. `response.created` → `response.in_progress` → `response.output_item.added` → `response.content_part.added`
4. Claude SDK chunks → `extract_stream_event_delta()` → `response.output_text.delta` (반복)
5. `response.output_text.done` → `response.content_part.done` → `response.output_item.done`
6. `response.completed` (usage 포함)

### MVP Event Sequence (8 types)
```
event: response.created
data: {"response": {id, object, status: "in_progress", ...}, "sequence_number": 0}

event: response.in_progress
data: {"response": {id, object, status: "in_progress", ...}, "sequence_number": 1}

event: response.output_item.added
data: {"output_index": 0, "item": {id, type: "message", role: "assistant", status: "in_progress"}, "sequence_number": 2}

event: response.content_part.added
data: {"item_id": "msg_*", "output_index": 0, "content_index": 0, "part": {type: "output_text", text: ""}, "sequence_number": 3}

event: response.output_text.delta          (repeated)
data: {"item_id": "msg_*", "output_index": 0, "content_index": 0, "delta": "Hello", "sequence_number": N}

event: response.output_text.done
data: {"item_id": "msg_*", "output_index": 0, "content_index": 0, "text": "Hello world", "sequence_number": N+1}

event: response.content_part.done
data: {"item_id": "msg_*", "output_index": 0, "content_index": 0, "part": {type: "output_text", text: "Hello world"}, "sequence_number": N+2}

event: response.output_item.done
data: {"output_index": 0, "item": {id, type: "message", status: "completed", content: [...]}, "sequence_number": N+3}

event: response.completed
data: {"response": {id, status: "completed", output: [...], usage: {...}}, "sequence_number": N+4}
```

## Implementation Plan

### Step 1: SSE 포맷 빌더 수정 (`streaming_utils.py`)
- **Actions**:
  - `make_response_sse()` 수정: `data: {"event": ...}` → `event: <type>\ndata: <json>\n\n`
  - `sequence_number` 파라미터 추가
  - 이벤트 타입별 올바른 필드명 사용 (`item`, `part`, `item_id`, `output_index`, `content_index`)
- **Deliverables**: 스펙 준수 SSE 빌더 함수
- **Dependencies**: 없음

### Step 2: Response 스트리밍 모델 추가 (`response_models.py`)
- **Actions**:
  - `ResponseContentPart` 별도 정의 불필요 — 기존 `ContentPart` 사용하되, 필드 `part` key로 직렬화
  - `ResponseObject`에 `usage` 기본값 처리 확인
  - `ResponseErrorDetail` 모델은 기존 것 활용
- **Deliverables**: 스트리밍 이벤트에 필요한 모델 정비
- **Dependencies**: Step 1

### Step 3: 스트리밍 오케스트레이터 수정 (`streaming_utils.py`)
- **Actions**:
  - `stream_response_chunks()` 수정:
    - 이벤트명 수정: `response.content_part.delta` → `response.output_text.delta`
    - `response.in_progress` 이벤트 추가
    - `response.done` → `response.completed`
    - `sequence_number` 카운터 추가 (0부터 단조 증가)
    - `item_id`, `output_index=0`, `content_index=0` 필드 추가
    - `data: [DONE]` 제거
    - 에러 시 `response.failed` 이벤트 전송
  - `response.output_text.done` 이벤트에 최종 텍스트 포함
  - `response.completed` 이벤트에 usage 포함
- **Deliverables**: 스펙 준수 8-이벤트 스트리밍 오케스트레이터
- **Dependencies**: Step 1, 2

### Step 4: 엔드포인트 핸들러 수정 (`main.py`)
- **Actions**:
  - `_run_stream()` 제너레이터에 usage 추적 추가
  - 스트리밍 완료 후 세션 업데이트 로직 유지
  - `response_registry` 등록 타이밍 검증 (스트리밍 시작 전 등록 — 현재 코드 이미 OK)
  - 에러 처리: SDK 에러 시 `response.failed` 이벤트 전송 후 스트림 종료
- **Deliverables**: 스트리밍 엔드포인트 핸들러 완성
- **Dependencies**: Step 3

### Step 5: 통합 검증
- **Actions**:
  - curl로 SSE 와이어 포맷 검증 (`event:` + `data:` 라인 확인)
  - 스트리밍 최종 텍스트 == 비스트리밍 텍스트 검증 (parity test)
  - `previous_response_id` 체이닝 검증
  - 에러 케이스 (SDK 실패, 빈 응답) 검증
- **Deliverables**: 동작 확인된 MVP 스트리밍
- **Dependencies**: Step 4

## Technical Considerations

- Claude SDK `extract_stream_event_delta()`는 thinking 델타도 반환하는데, MVP에서는 thinking을 텍스트에 포함시킬지 결정 필요 (현재 chat과 동일하게 `<think>` 태그로 포함)
- `sequence_number`는 함수 내 로컬 카운터로 관리 (단일 스트림 내 단조 증가)
- usage는 `response.completed`에만 포함 (MVP 동작, OpenAI 스펙 준수)
- `ResponseObject`의 `model_dump()` 시 `usage` 기본값이 `{input_tokens: 0, output_tokens: 0}`인데, 스트리밍 완료 시점에 실제 값으로 업데이트 필요
- session lock은 클라이언트 생성 시에만 필요, 스트리밍 중에는 불필요 (현재 코드 이미 OK)

## Risk Management

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| SSE 포맷 파싱 실패 (OpenAI SDK) | High | Medium | 공식 스펙 기반 구현 + curl 와이어 검증 |
| 스트리밍/비스트리밍 결과 불일치 | High | Low | 동일 accumulator 패턴 사용, parity test |
| 중단된 스트림에서 response.completed 전송 | Medium | Low | try/finally로 상태 관리, 에러 시 response.failed |
| sequence_number 불연속 | Low | Low | 단순 카운터로 구현, 분기 없음 |
| previous_response_id 스트리밍 시 세션 오염 | Medium | Low | 스트리밍 완료 후에만 세션 업데이트 (현재 코드 이미 OK) |
